# 00-Project-Vision

> สถานะ: Draft v1

## 1. ภาพรวม

EduQuest Online คือเกม MMORPG เต็มรูปแบบที่มีระบบการเรียนแทรกอยู่แบบแนบเนียน ไม่ใช่เกมทำข้อสอบ และไม่ใช่แอปเรียนที่ใส่กราฟิกเกมเข้าไป แต่เป็นโลกแฟนตาซีออนไลน์ที่ผู้เล่นออกสำรวจ ต่อสู้ เก็บของ คราฟของ ลงดันเจียน เข้ากิลด์ ล่าบอส และแข่งขันกับผู้เล่นอื่น โดยมี "ความรู้" เป็นหนึ่งในพลังหลักของตัวละคร

เป้าหมายสำคัญคือทำให้ผู้เล่นรู้สึกว่าเขากำลังเล่นเกม MMORPG จริง แต่ทุกการเติบโตในเกมค่อย ๆ ผลักให้ผู้เล่นเรียนรู้ ฝึกคิด และสะสมความชำนาญในวิชาต่าง ๆ ไปพร้อมกัน

## 2. หลักการออกแบบ

- Game First: เกมต้องสนุกก่อน ระบบเรียนรู้ต้องเสริมประสบการณ์ ไม่ขัดจังหวะการเล่น
- Learning as Power: ความรู้ต้องมีผลกับพลัง การปลดล็อก ความแม่นยำ โอกาสสำเร็จ และทางเลือกของผู้เล่น
- Seamless Quiz: คำถามต้องถูกนำเสนอเป็นส่วนหนึ่งของ action เช่น โจมตี ใช้เวท เปิดหีบ เจรจา คราฟ หรือแก้กับดัก
- MMORPG Depth: เกมต้องมีระบบหลักครบแบบ MMORPG ได้แก่ world map, class, stat, item, quest, dungeon, boss, party, guild, economy และ PvP
- Realtime Tile World: แผนที่โลกเป็นช่อง ๆ แบบออนไลน์ ผู้เล่นเดิน เจอกัน ต่อสู้ และยึดพื้นที่กันได้แบบ realtime
- Territory War: การครอบครองพื้นที่เป็นแรงขับเคลื่อนหลักของโลก ผู้เล่นและเมืองเกิดสามารถขยายอิทธิพลผ่านการตีพื้นที่ว่างหรือชนะผู้เล่นที่ยืนป้องกันพื้นที่
- Fair Progression: ผู้เล่นที่เก่งเกมและผู้เล่นที่ตั้งใจเรียนรู้ต้องมีทางเติบโตได้ทั้งคู่
- Long-Term Mastery: ความรู้แต่ละหมวดต้องสะสมเป็น Knowledge Mastery เพื่อปลดล็อกสกิล อุปกรณ์ เควส และ achievement
- Social Learning: ปาร์ตี้ กิลด์ และ raid ควรทำให้ผู้เล่นช่วยกันคิด ช่วยกันตอบ และวางแผนร่วมกัน

## 3. ตัวตนของเกม

เกมนี้ควรรู้สึกเหมือน:

- Ragnarok / MapleStory / World of Warcraft ในแง่ความเป็น MMORPG
- Pokemon / Monster Hunter ในแง่การสะสม ไล่ล่า และเติบโตระยะยาว
- Kahoot / Quiz RPG ในแง่การใช้คำถาม แต่ต้องไม่ให้หน้าตาเหมือนห้องสอบ
- Sandbox world map ที่ผู้เล่นมีอิสระเลือกเป้าหมายของตัวเอง

เกมนี้ไม่ควรรู้สึกเหมือน:

- แอปทำแบบฝึกหัด
- ระบบสอบออนไลน์
- เกมที่ทุก action ถูกหยุดด้วยหน้าคำถามยาว ๆ
- เกมเด็กเกินไปจนผู้เล่น MMORPG จริงไม่อยากเล่น

## 4. Core Promise

ผู้เล่นเข้ามาเล่นเพราะอยากผจญภัยในโลก MMORPG และอยู่ต่อเพราะรู้สึกว่าตัวละครของเขาเก่งขึ้นจากทั้งการเล่น การวางแผน และความรู้ที่สะสมจริง

ประโยคแกนของโปรเจกต์:

> "เล่นให้สนุกเหมือน MMORPG เรียนรู้โดยไม่รู้สึกว่ากำลังถูกบังคับเรียน"

## 5. กลุ่มผู้เล่นเป้าหมาย

Primary Target:

- นักเรียนระดับประถมปลายถึงมัธยม
- ผู้เล่นที่ชอบเกม RPG, MMORPG, เก็บเลเวล, สะสมของ และเล่นกับเพื่อน
- ผู้ปกครองหรือครูที่อยากให้เด็กใช้เวลาเล่นเกมแล้วได้ฝึกความรู้ไปด้วย

Secondary Target:

- ผู้เล่นทั่วไปที่ชอบเกม world map, idle progression, crafting, boss hunt และ collection
- ผู้เรียน Coding หรือภาษาอังกฤษที่อยากฝึกผ่านเกม

## 6. เสาหลักของเกม

### 6.1 Adventure

โลกต้องชวนให้สำรวจ มี biome, เมือง, ดันเจียน, บอส, event และพื้นที่ลับ ผู้เล่นควรรู้สึกว่าแผนที่มีสิ่งให้ค้นหาเสมอ แผนที่เป็นระบบ tile ที่ผู้เล่นเดินทีละช่อง เห็นผู้เล่นอื่นบนโลกเดียวกัน และมีโซนที่สามารถแย่งชิงพื้นที่ได้

### 6.2 Combat

การต่อสู้ต้องเป็นระบบ RPG realtime/near-realtime ที่มี stat, element, skill, equipment, buff/debuff และ strategy คำถามเป็นตัวเพิ่มหรือลดประสิทธิภาพ ไม่ใช่สิ่งเดียวที่ตัดสินผลแพ้ชนะ ผู้เล่นสามารถต่อสู้กับมอนสเตอร์และผู้เล่นอื่นเพื่อป้องกันหรือยึดพื้นที่

### 6.3 Knowledge

ความรู้เป็นระบบ progression ระยะยาว แยกตามหมวด เช่น Math, Science, Thai, English, History, Geography, Logic และ Coding ค่าความชำนาญนี้มีผลต่อการปลดล็อกและโบนัสเฉพาะทาง

### 6.4 Collection

ผู้เล่นควรสะสมอุปกรณ์ สัตว์เลี้ยง mount วัตถุดิบ recipe title achievement และของหายากได้จำนวนมาก เพื่อสร้างแรงจูงใจระยะยาว

### 6.5 Social

เกมต้องรองรับ party, guild, chat, guild quest, guild boss, raid, trading, marketplace และสงครามยึดพื้นที่ เพื่อให้โลกมีชีวิตจากผู้เล่น ไม่ใช่แค่ระบบคนเดียว

### 6.6 Live World

โลกควรมี daily quest, weekly quest, world event, world boss, season ranking และ content หมุนเวียน เพื่อให้ผู้เล่นมีเหตุผลกลับมาเล่น

## 7. แนวทางการใช้คำถามในเกม

คำถามควรถูกใช้เป็น "จังหวะตัดสินใจ" มากกว่า "ด่านสอบ"

ตัวอย่าง:

- เดินเข้าพื้นที่อันตราย: ตอบถูกเพื่อลด stamina cost หรือหลบกับดัก
- โจมตีมอนสเตอร์: ตอบถูกเพื่อเพิ่ม damage, critical chance หรือ skill success
- คราฟของ: ตอบถูกเพื่อเพิ่ม success rate หรือเพิ่ม bonus stat
- เปิดหีบ: ตอบถูกเพื่อเพิ่มโอกาสได้ของ rare
- คุยกับ NPC: ตอบถูกเพื่อปลด dialogue, clue หรือ quest path
- Raid boss: สมาชิกปาร์ตี้ช่วยกันตอบ mechanic สำคัญระหว่างสู้
- Movement: ตอบถูกเพื่อเดินได้ไกลขึ้น เช่น ปกติเดิน 1 ช่อง ตอบถูกเดินได้ 2 ช่อง และสัตว์ขี่เพิ่มระยะเดินตามชนิด
- Territory Claim: ตอบถูกเพื่อเพิ่มความเร็วหรือโอกาสสำเร็จในการยึดพื้นที่ว่าง

หลักสำคัญคือผู้เล่นยังเล่นเกมได้แม้ตอบผิด แต่คำตอบที่ถูกต้องควรให้รางวัลชัดเจน

## 8. โครงสร้างโลกออนไลน์เบื้องต้น

แผนที่โลกเป็น grid/tile ขนาดใหญ่ มีจุดศูนย์กลางอยู่กลางแผนที่ และมีเมืองเกิดหลัก 4 เมืองอยู่รอบทิศเหนือ ใต้ ตะวันออก และตะวันตก ผู้เล่นเลือกเมืองเกิดตอนสร้างตัวละคร หรือให้ระบบสุ่มเมืองเกิดได้

ลำดับพื้นที่จากเมืองเกิด:

1. เมืองเกิดและ safe zone
2. พื้นที่ฟาร์มเริ่มต้นรอบเมือง
3. พื้นที่ทรัพยากรและมอนสเตอร์ระดับกลาง
4. พื้นที่ยึดครอง/แย่งชิง
5. พื้นที่กลางแผนที่ที่มีทรัพยากร บอส และรางวัลสูง

กฎหลัก:

- ผู้เล่นของแต่ละเมืองสามารถขยายพื้นที่จากเมืองตัวเองไปยังพื้นที่รอบนอกได้
- พื้นที่ว่างสามารถถูกยึดได้ด้วยการตีพื้นที่หรือทำ action claim
- ถ้ามีผู้เล่นอื่นยืนป้องกันอยู่ ต้องชนะผู้เล่นนั้นก่อนจึงยึดพื้นที่ได้
- เมื่อผู้เล่นตาย จะกลับไปเกิดที่เมืองเกิดของตัวเอง
- เมืองเกิดควรเป็น safe zone ไม่ถูกยึดถาวรในช่วงแรกของการออกแบบ
- พื้นที่กลางแผนที่ควรเป็นเป้าหมายสำคัญระยะยาวของสงครามระหว่างเมือง

## 9. ขอบเขต MVP แรก

MVP แรกควรพิสูจน์ว่า loop หลักสนุกจริงก่อนขยายระบบใหญ่

ระบบที่ควรมี:

- Login แบบง่าย
- สร้างตัวละคร
- เลือกเมืองเกิด 1 ใน 4 ทิศ
- World map แบบ tile
- Movement ด้วยพลังงานที่เชื่อมกับคำถาม
- ผู้เล่นเห็นตำแหน่งกันบนแผนที่แบบ realtime
- Monster encounter
- Combat พื้นฐาน
- PvP พื้นฐานบน tile
- Territory claim พื้นฐาน
- Quiz engine พื้นฐาน
- Knowledge Mastery พื้นฐาน
- Inventory เบื้องต้น
- Item drop เบื้องต้น
- Save/load player progression

ระบบที่ยังไม่จำเป็นใน MVP:

- Guild เต็มระบบ
- PvP
- Auction House
- Raid ขนาดใหญ่
- Pet evolution
- Season ranking
- AI question generation แบบเต็มรูปแบบ

## 10. แนวทางเทคนิคระดับสูง

Frontend:

- Next.js สำหรับ web app, account, UI, dashboard และ shell หลัก
- Phaser หรือ PixiJS สำหรับ world map และ gameplay scene

Backend:

- NestJS สำหรับ API และ game service
- WebSocket หรือ Colyseus สำหรับ realtime gameplay

Database:

- PostgreSQL สำหรับข้อมูลถาวร เช่น player, character, item, quest, knowledge
- Redis สำหรับ session, cache, matchmaking, realtime state บางส่วน และ cooldown

AI:

- ใช้ OpenAI API เพื่อช่วยสร้างคำถาม คำอธิบายคำตอบ dialogue และ content draft
- คำถามที่ AI สร้างควรผ่าน moderation/approval ก่อนใช้กับผู้เล่นจริง

## 11. เกณฑ์วัดความสำเร็จ

- ผู้เล่นรู้สึกว่าเกมสนุกแม้ไม่สนใจเรื่องการเรียนในตอนแรก
- ผู้เล่นตอบคำถามซ้ำได้โดยไม่รู้สึกว่าถูกบังคับสอบ
- Knowledge Mastery ทำให้ผู้เล่นอยากกลับไปฝึกวิชาที่ตัวเองอ่อน
- ระบบต่อสู้และสำรวจยังสนุกแม้ตัดคำถามออกชั่วคราว
- ผู้เล่นมีเป้าหมายระยะสั้น กลาง และยาวชัดเจน
- ผู้เล่นเข้าใจทันทีว่าโลกนี้คือแผนที่ออนไลน์ที่แย่งชิงพื้นที่กันได้
- การยึดพื้นที่สร้างแรงจูงใจให้ผู้เล่นออกจากเมืองและร่วมมือกับฝ่ายเดียวกัน
- ครูหรือผู้ปกครองสามารถเห็นคุณค่าการเรียนรู้โดยไม่ทำลายความสนุกของเกม

## 12. ความเสี่ยงหลัก

- คำถามถี่เกินไปจนจังหวะเกมสะดุด
- ระบบเรียนเด่นเกินจนผู้เล่นรู้สึกเหมือนทำข้อสอบ
- ระบบ MMORPG กว้างเกิน ทำให้ MVP ใหญ่เกินควบคุม
- PvP อาจไม่ยุติธรรมหากคำถามหรือ Knowledge Mastery ให้พลังมากเกิน
- ผู้เล่นใหม่อาจถูกผู้เล่นเก่าดักตีใกล้เมืองเกิด ต้องมี safe zone และ protection
- เมืองใดเมืองหนึ่งอาจ snowball ยึดพื้นที่มากเกินไป ต้องมีระบบ balance และ comeback
- AI สร้างคำถามผิดหรือคำตอบคลุมเครือ
- เศรษฐกิจเกมเสียสมดุลจาก item drop, trading หรือ marketplace
